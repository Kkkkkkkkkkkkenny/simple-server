# simple-server 原理讲解

本文从底层网络模型出发，逐步拆解 `simple-server` 背后的原理。

---

## 1. 从 URL 说起

当你在浏览器输入 `http://192.168.1.5:8080/index.html` 并回车时，实际发生了这几件事：

1. 浏览器从 URL 中解析出**协议**（http）、**目标 IP**（192.168.1.5）、**端口**（8080）、**路径**（/index.html）
2. 通过 ARP 协议找到目标 IP 对应的物理网卡地址（MAC）
3. 建立 TCP 连接到 `192.168.1.5:8080`
4. 发送 HTTP 请求报文（GET /index.html）
5. 服务器解析 HTTP 请求，读取文件，返回 HTTP 响应
6. 浏览器解析响应，渲染页面

simple-server 做的事情就是**第 5 步**——解析 HTTP 请求并返回文件内容。

---

## 2. IP 地址与端口

### 2.1 127.0.0.1（localhost）

每个操作系统都有一个虚拟回环网卡，IP 固定为 `127.0.0.1`。发往 `127.0.0.1` 的数据包不会离开本机，直接在内核层面就被转回。所以服务监听 `127.0.0.1` 时，只有本机能访问。

### 2.2 0.0.0.0

`0.0.0.0` 是一个特殊地址，表示"本机的所有 IP 地址"。当服务绑定到 `0.0.0.0` 时，它同时监听所有网卡——回环网卡（127.0.0.1）、以太网卡（192.168.x.x）、Tailscale 虚拟网卡（100.x.x.x）等。

```
服务绑定 127.0.0.1:8080  →  只有本机 localhost 能访问
服务绑定 0.0.0.0:8080    →  所有网络接口均可访问
```

这也是 simple-server 默认使用 `0.0.0.0` 的原因——让局域网和 Tailscale 的流量都能到达服务。

### 2.3 端口

端口是一个 16 位的整数（0–65535），用于区分同一台机器上的不同服务。比如：

- `:22` — SSH
- `:80` — HTTP（默认）
- `:443` — HTTPS（默认）
- `:8080` — HTTP 备用端口（simple-server 默认）

## 3. 局域网通信

### 3.1 子网与广播

办公室/家里的设备通常连接在同一个路由器下，形成一个**局域网**（LAN）。路由器的 DHCP 服务会给每个设备分配一个私有 IP：

- `192.168.0.x / 192.168.1.x`
- `172.16–31.x.x`
- `10.x.x.x`

这些私有 IP 在公网不可路由，只能在局域网内通信。

### 3.2 局域网内如何访问

当设备 A（`192.168.1.2`）访问设备 B（`192.168.1.5:8080`）时：

1. A 发现目标 IP `192.168.1.5` 在同一子网（通过子网掩码判断）
2. A 发送 ARP 广播："谁有 192.168.1.5？"
3. B 回复："我有，我的 MAC 地址是 xx:xx:xx:xx:xx:xx"
4. A 直接向 B 的 MAC 地址发送数据包
5. 数据包经过交换机 → 到达 B → 到达 B 的 8080 端口上的服务

整个过程不经过路由器公网，速度接近物理带宽上限。

## 4. NAT 与端口转发

### 4.1 为什么需要 NAT

IPv4 地址已经耗尽，你的路由器只有一个公网 IP，但家里可能有十几台设备。NAT（Network Address Translation）技术让多台设备共享一个公网 IP：

```
公网互联网
    │
路由器（公网 IP: 203.0.113.5）
    │
    ├── 192.168.1.2（你的笔记本）
    ├── 192.168.1.3（手机）
    └── 192.168.1.5（WSL2 主机）
```

NAT 会维护一张映射表，把出站连接的公网端口映射到内网设备的端口。返回的数据包根据这张表找到正确的内网设备。

### 4.2 端口转发

NAT 默认只允许**出站**连接。如果想从公网访问内网的服务，需要配置端口转发（Port Forwarding）：

```
公网用户访问 203.0.113.5:8080
    │
路由器端口转发规则：
  公网 0.0.0.0:8080 → 192.168.1.5:8080
    │
到达 Windows 主机 192.168.1.5:8080
```

在 WSL2 场景下，Windows 充当了"路由器"的角色，WSL2 是内网。所以需要额外的端口转发：

```
其他设备 192.168.1.X:8080
    │
Windows 端口转发（netsh）：
  0.0.0.0:8080 → 172.28.184.5:8080
    │
WSL2 内的 simple-server（172.28.184.5:8080）
```

对应的 netsh 命令：
```powershell
netsh interface portproxy add v4tov4 listenaddress=0.0.0.0 listenport=8080 connectaddress=172.28.184.5 connectport=8080
```

## 5. Tailscale 原理

### 5.1 问题

端口转发配置麻烦，而且：
- 没有公网 IP 就无法工作（手机网络、公司内网）
- 暴露服务到公网有安全风险
- 设备移动后 IP 变化，配置要跟着改

### 5.2 解决方案：WireGuard + 协调服务器

Tailscale 基于 **WireGuard** 协议实现了一个零配置的虚拟专用网络（VPN）。

**WireGuard** 的工作原理：

1. 每台设备生成一对公私钥
2. 设备向 Tailscale 的协调服务器注册公钥
3. 协调服务器告诉每台设备："你在 100.81.74.3，另外那台在 100.81.74.42"
4. 两台设备之间直接建立加密隧道（Tunnel），通信不经过协调服务器

```
设备 A（100.81.74.3）                   设备 B（100.81.74.42）
    │                                          │
    └──────────── WireGuard 加密隧道 ───────────┘
                        │
              不经过公网路由，直接 P2P 连接
```

如果两台设备在同一个局域网内，Tailscale 会自动走局域网直连（延迟最低）。如果跨网络，会通过 NAT 穿透技术建立 UDP 直连。

### 5.3 Tailscale IP（100.x.x.x）

Tailscale 分配的 `100.x.x.x` 是一个虚拟 IP，它在 WireGuard 隧道中路由。当设备 B 访问 `100.81.74.3:8080` 时：

1. 操作系统看到目标为 `100.x.x.x`，走 WireGuard 虚拟网卡
2. WireGuard 加密数据包，发送到设备 A 的真实 IP（192.168.x.x 或公网 IP）
3. 设备 A 的 WireGuard 解密数据包，交给 simple-server

整个过程对 simple-server 完全透明——它看到的源 IP 是 Tailscale 虚拟 IP，但它不关心这些，它只负责返回 HTTP 响应。

## 6. WSL2 网络模型

### 6.1 NAT 模式（默认）

```
Windows 主机（192.168.1.5）
    │
    ├── WSL2 虚拟机（172.28.184.5）
    │       └── simple-server（:8080）
    │
    └── 其他局域网设备（192.168.1.X）
```

WSL2 运行在 Hyper-V 虚拟机中，使用 NAT 网络。Windows 会动态给 WSL2 分配一个私有 IP（通常是 `172.x.x.x`），其他局域网设备不知道这个地址的存在。

**Windows 到 WSL2**：Windows 可以通过 `172.28.184.5:8080` 直接访问（Windows 知道路由）。
**其他设备到 WSL2**：其他设备不知道 `172.x.x.x` 是哪里，所以需要 Windows 的端口转发。

### 6.2 镜像网络模式（WSL 2.0+）

在 `%USERPROFILE%\.wslconfig` 中配置：

```ini
[wsl2]
networkingMode=mirrored
```

这种模式下，WSL2 直接复用 Windows 的网卡，WSL2 里的服务在 Windows 的 IP 上直接可用，不需要端口转发。但镜像模式目前可能有兼容性问题，部分网络功能受限。

## 7. HTTP 协议基础

### 7.1 请求-响应模型

HTTP 是一个请求-响应协议：

```
客户端 → 服务器: GET /index.html HTTP/1.1
                 Host: 192.168.1.5:8080

服务器 → 客户端: HTTP/1.1 200 OK
                 Content-Type: text/html
                 Content-Length: 1234

                 <!DOCTYPE html>
                 <html>...
```

### 7.2 Python http.server 做了什么

```
浏览器连接 :8080
    │
socket.accept() → 建立 TCP 连接
    │
recv() → 读取 HTTP 请求报文
    │
解析请求行: GET /index.html HTTP/1.1
解析请求头: Host, User-Agent...
    │
路径映射到文件系统: /index.html → ./index.html
    │
读取文件 → 构造 HTTP 响应 → send() 返回
    │
关闭连接（或复用）
```

simple-server 就是对这个流程的封装，加上 IP 自动检测、友好启动信息等便利功能。

## 总结

| 概念 | 类比 |
|------|------|
| `127.0.0.1` | 自己对自己说话 |
| `0.0.0.0` | 对所有门喊话 |
| 端口 | 门牌号，区分不同服务 |
| NAT | 公司前台，转发外部来电 |
| 端口转发 | 前台登记：打 X 号找 Y |
| Tailscale | 内部专线电话网，不管你在哪 |
| WSL2 NAT | 虚拟机躲在 Windows 后面 |
